Add the `StripeEvent` idempotency model at the bottom of the tenant-scoped section. Note: this table is written by the webhook handler BEFORE any clinic context is resolved, so the PK is `event.id` (Stripe's own ID, which is globally unique).
```prisma
model StripeEvent {
  id                  String   @id                          // Stripe event.id, e.g. evt_1AbCdEf…
  type                String                                 // e.g. account.updated
  connectedAccountId  String?                                // event.account — null for platform events
  apiVersion          String?                                // event.api_version
  payload             Json                                   // full event object for forensic replay
  receivedAt          DateTime @default(now())
  processedAt         DateTime?                              // null = in-flight or failed
  processingError     String?                                // last error message if processing failed

  @@index([connectedAccountId])
  @@index([type, receivedAt])
  @@index([processedAt])
}
```

Do NOT add a clinic FK on StripeEvent. A connectedAccount may arrive in a webhook before the Clinic is fully linked, and we want the event log to survive clinic deletions for audit.

**Step 1 — Write `prisma/sql/002-stripe-rls.sql`.**

StripeEvent must be RLS-enabled so production queries don't accidentally leak event bodies across tenants. BUT the webhook handler ingests events BEFORE resolving a clinic_id (it only has Stripe's `event.account`), so the policy must permit inserts with the GUC unset. Mirror the existing Clinic two-mode pattern:

```sql
-- Phase 2 RLS — Stripe Connect event idempotency store.
-- Two-mode policy identical to Clinic (see 001-rls-policies.sql):
--   (a) GUC unset  → permissive (webhook ingest before clinic resolution)
--   (b) GUC set    → strict (dashboard + background jobs reading own events)
-- Scoping is by connectedAccountId → Clinic.stripeAccountId at query time.

ALTER TABLE "StripeEvent" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "StripeEvent" FORCE ROW LEVEL SECURITY;
DROP POLICY IF EXISTS tenant_isolation ON "StripeEvent";
CREATE POLICY tenant_isolation ON "StripeEvent"
  USING (
    NULLIF(current_setting('app.current_clinic_id', true), '') IS NULL
    OR "connectedAccountId" IN (
      SELECT "stripeAccountId" FROM "Clinic"
      WHERE id = NULLIF(current_setting('app.current_clinic_id', true), '')::uuid
    )
  )
  WITH CHECK (
    NULLIF(current_setting('app.current_clinic_id', true), '') IS NULL
    OR "connectedAccountId" IN (
      SELECT "stripeAccountId" FROM "Clinic"
      WHERE id = NULLIF(current_setting('app.current_clinic_id', true), '')::uuid
    )
  );
```

**Step 3 — Create `src/lib/stripe/types.ts`.**

Export the types and the publish-gating helper. This is the contract Plans 02-02 and 02-03 import — define it here so downstream tasks don't re-derive it.

```typescript
import type { Prisma } from '@prisma/client';

/**
 * Subset of Stripe's Account.requirements we care about.
 * See https://docs.stripe.com/api/accounts/object#account_object-requirements
 *
 * NOT an exhaustive mapping — just the fields PUB-02 gates on and the
 * dashboard banner displays.
 */
export interface StripeConnectRequirements {
  currently_due: string[];
  eventually_due: string[];
  past_due: string[];
  pending_verification: string[];
  disabled_reason: string | null;
}

/**
 * Mirror of the Prisma OnboardingState enum as a TS string union.
 * Kept in lockstep with prisma/schema.prisma → enum OnboardingState.
 */
export type OnboardingState =
  | 'not_started'
  | 'in_progress'
  | 'action_required'
  | 'complete'
  | 'restricted';

/**
 * Snapshot of Connect capability state as persisted on Clinic.
 * Built from stripe.accounts.retrieve() OR account.updated event.
 */
export interface ConnectSnapshot {
  stripeAccountId: string;
  chargesEnabled: boolean;
  payoutsEnabled: boolean;
  detailsSubmitted: boolean;
  disabledReason: string | null;
  requirements: StripeConnectRequirements;
}

/**
 * PUB-02 gate: publish is allowed IFF all three conditions hold.
 * This is the canonical predicate — used in publish server action
 * AND dashboard UI to keep the Publish button state consistent.
 */
export function isPublishReady(snapshot: {
  chargesEnabled: boolean;
  payoutsEnabled: boolean;
  disabledReason: string | null;
}): boolean {
  return (
    snapshot.chargesEnabled === true &&
    snapshot.payoutsEnabled === true &&
    snapshot.disabledReason === null
  );
}

/**
 * Derive OnboardingState from a snapshot — single source of truth used by
 * the webhook handler when persisting + the UI when rendering status.
 */
export function deriveOnboardingState(args: {
  stripeAccountId: string | null;
  chargesEnabled: boolean;
  payoutsEnabled: boolean;
  detailsSubmitted: boolean;
  disabledReason: string | null;
  requirementsCurrentlyDue: string[];
}): OnboardingState {
  if (!args.stripeAccountId) return 'not_started';
  if (args.disabledReason) return 'restricted';
  if (isPublishReady(args)) return 'complete';
  if (args.requirementsCurrentlyDue.length > 0) return 'action_required';
  return 'in_progress';
}

/**
 * Prisma-side type helper: the JSON shape stored in Clinic.stripeRequirements.
 * Use `as Prisma.InputJsonValue` when writing, cast via this type when reading.
 */
export type StripeRequirementsJson = StripeConnectRequirements & Prisma.JsonObject;
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Webhook handler → StripeEvent insert | Webhook runs BEFORE clinic_id is resolved; RLS policy must permit GUC-unset writes. |
| Dashboard query → StripeEvent read | Must be scoped to the clinic's own connectedAccountId (never return another clinic's event log). |
| Schema push → production DB | `prisma db push` on a multi-tenant DB with existing rows must not silently drop or alter tenant data. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | Information Disclosure | StripeEvent.payload (contains full Stripe event, may include owner PII) | mitigate | RLS two-mode policy scopes dashboard reads to clinic's own `connectedAccountId`; GUC-unset permissive only for webhook ingest path (server-only, never user-scoped). |
| T-02-01-02 | Tampering | Clinic.stripeChargesEnabled flag | mitigate | Only the webhook handler (Plan 02-03) writes these fields; UI routes never mutate them. Documented in schema comment. Verification: grep for direct `.update({ stripeChargesEnabled })` calls from non-webhook code must return zero hits post-phase. |
| T-02-01-03 | Repudiation | Whether Stripe "really" sent a given capability flip | accept | `StripeEvent.payload` stores the raw event body + `apiVersion`; signature verification (Plan 02-02) guarantees authenticity at ingest; no further non-repudiation needed for v1 demo. |
| T-02-01-04 | Denial of Service | StripeEvent table bloat under replay storms | accept | PK is `event.id`; duplicate inserts no-op at the unique constraint — storage cost is one row per real event. Retention policy deferred to post-v1. |
| T-02-01-05 | Elevation of Privilege | RLS permissive GUC-unset mode | mitigate | Permissive-when-unset is identical to the Clinic pattern accepted in Phase 1 (01-03 SUMMARY decisions). Webhook route is server-only; no path exposes an unscoped connection to user input. |
| T-02-01-06 | Spoofing | Clinic with `stripeAccountId=null` but stripeChargesEnabled=true (impossible state) | mitigate | Defaults are `false`/`null`. UI gates on `isPublishReady()` which checks charges+payouts+!disabled — a stripeAccountId=null clinic still has chargesEnabled=false, so no impossible-state write path exists in this plan. |
</threat_model>
